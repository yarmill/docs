Results answers the closing question of the training loop — did the work produce the effect?
It keeps the outcomes — races, matches, and structured tests — in a configurable table per
athlete, so performance over a season is data, not memory.

**For:** coaches and athletes · **Where:** Web app

<Info>Results tables are **configured per team and sport**. Which columns exist, the fields
on the form, their types, and the dropdown values are all set for your sport — the columns
shown below are one team's example, not universal. Some sports **import results
automatically** from a federation database (for example IBU or FIS) instead of entering races
by hand; see [Automatic import](#automatic-import-federation-databases).</Info>

## The results table

Results is a **per-athlete** module: pick an athlete from the group panel first, and their
table loads. The page shows a configured title (for this team, **"Races results"**), a **New
result** button top-right, and a table filling the page — one row per result.

For a biathlon team, the configured columns are **Date · Location · Weather · Discipline ·
Rank · Note**. A filled row reads, for example: **24/06/2025 · Zermatt · ☀️ sunny ·
Slopestyle · 1. · Top** — where the **Weather** value comes from a dropdown and renders with
an emoji. Your sport's columns will differ.

Each row carries inline **edit** (pencil) and **delete** (trash) icons at its right end. There
is no separate detail view — editing opens the same form as **New result**, populated with the
row's values.

Before any results exist for an athlete, the table reads **"No records"** rather than showing
an empty grid.

## Record a result

<Steps>
  <Step title="Select the athlete">
    Open **Results** and choose an athlete from the group panel — results are kept per person.
  </Step>
  <Step title="Open the form">
    Select **New result**. A full-screen form titled **"New result"** opens with the configured
    fields.
  </Step>
  <Step title="Fill in the fields">
    Enter the result. For this team's setup, that's:

    <ParamField path="Date" required>
      A date field. **Required**, and it defaults to today.
    </ParamField>
    <ParamField path="Location">
      Free text. Optional.
    </ParamField>
    <ParamField path="Weather">
      A dropdown of configured values. Optional. Values render with an emoji in the table.
    </ParamField>
    <ParamField path="Discipline">
      Free text in this instance. Optional. A different sport's setup may make this a dropdown.
    </ParamField>
    <ParamField path="Rank">
      Free text, labelled **Final ranking**. Optional.
    </ParamField>
    <ParamField path="Note">
      Free-text notes. Optional.
    </ParamField>

    Only **Date** is required; every other field is optional, and which fields appear is
    configured for your sport.
  </Step>
  <Step title="Save">
    Select **Save result** to write the row, or **Cancel** to discard. The form is **explicit
    save** — opening it creates nothing, and the result is recorded only when you save.
  </Step>
</Steps>

To edit a result later, select the row's **edit** (pencil) icon — the same form reopens
populated — adjust the fields, and **Save result** again. To remove one, use the row's
**delete** (trash) icon.

## Find results fast

Every column header carries a **funnel** icon. Selecting it opens a popover that combines
sorting and a filter for that column:

- **Sort** at the top — **Ascending** or **Descending**.
- A **filter** below, matched to the column type. For the **Date** column, the options are
  **This week · Last week · This month · Last month · Exact date · Range**.
- **Clear** and **Apply** at the bottom.

Use the **Discipline** funnel to compare an athlete's sprint results, or the **Date** funnel
to scope a season block — without scrolling the whole table.

## Automatic import (federation databases)

Some sports don't enter races by hand at all. Where a federation publishes results in a public
database — **IBU SIWIDATA** for biathlon, **FIS** for skiing and snowboard, and others —
results can flow into the table automatically. The coach then reviews the table rather than
re-keying races.

<Info>Automatic import is **configured per sport** and depends on two things: the connection
set up under [Integrations](/en/platform/integrations) (which requires a license agreement
with the data provider), and the athlete's federation ID — for example their **IBU ID** — set
on their [Athlete profile](/en/platform/athlete-profile).</Info>

## Testing & Diagnostics

Teams that run structured testing — strength, lactate, body composition, or sport-specific
protocols — record it in the same kind of configurable form, a sibling tabular module
alongside Results. Each test feeds its reports in [Analytics](/en/analytics/analytics). If your
team uses them, you'll find them next to your results tables.

## How it connects

Results sits at the end of the loop **Plan → Reality → Results → Analytics**, recording the
outcome of the work logged in [Reality](/en/reality/training-log):

- Race and test data feed **race analyses** and **Test & Diagnostics reports** in
  [Analytics](/en/analytics/analytics), so the team reads progress against the field and
  against its own benchmarks.
- Federation connections for automatic import are set up under
  [Integrations](/en/platform/integrations).
- An athlete's external IDs — such as their **IBU ID** — on the
  [Athlete profile](/en/platform/athlete-profile) link them to the results database for import.
